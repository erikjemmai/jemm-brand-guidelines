Button hierarchy

Primary — solid emerald in light mode, neon in dark mode. Secondary — 1 px outline in the same green. Tertiary — neutral ghost. Link — green text, underline on hover; use a <button> or an <a>.

Secondary · medium · 44 px height Outlined in the same green · transparent fill
Tertiary · medium · 44 px height No border · neutral text
Link · text only Green · underline on hover · button or anchor

Placement & order

Button order follows reading direction and alignment. Primary always leads — but “lead” means leftmost on left-aligned CTAs, and rightmost on right-aligned action bars.

Left-aligned · hero, cards, marketing sections

Hero / card / website section · CTAs start from the left

Primary first, then Secondary, then Tertiary — emphasis decreases left → right.

Own your AI

Primary→Secondary→Tertiary

Right-aligned · table headers, modals, toolbars, form footers

Table header / modal footer / dense UI · actions flush right

Primary rightmost, Secondary to its left, Tertiary further left — emphasis increases right → left.

Devices

Primary←Secondary←Tertiary

Use .ds-btn-row--start for left-aligned groups · .ds-btn-row--end for right-aligned groups. DOM order should still be Tertiary → Secondary → Primary in right-aligned bars so keyboard tab order matches visual priority.

Size tiers

Tiers are defined by fixed height (32 / 44 / 56 px) — not label length. Width grows with copy. Use short labels so buttons stay scannable.

Large · hero / conversion 56 px height · 18 px bold · 14×28 px padding · 12 px radius

Button labels

Keep copy short. Height stays fixed; long labels widen the button and break layout rhythm. 1 word is ideal · 2 words is fine · 3 words is pushing it · 4+ — nope.

Ideal · 1 word

Clear, scannable, fits every tier.

OK · 2 words

Acceptable for common phrases — still prefer one word when you can (“Continue” over “Log in”).

Push it · 3 words

Only on Large buttons or wide layouts. Shrink copy first.

Nope · 4+ words

Never. Use a headline for context; the button gets one verb.

Photo, video & image backgrounds

Busy or multicolor backgrounds (photos, video, gradients, art) can make green buttons clash. Use white solid primary and white outline secondary instead — they stay readable on anything.

Simulated hero fill — green CTAs would fight the palette. White buttons stay neutral.

Apply .ds-btn--inverse and .ds-btn--inverse-outline only on media overlays — not on flat UI surfaces.

States

Default · Hover · Active (scale 0.98) · Focus-visible (2 px ring + offset) · Disabled (45% opacity). Tab to demo focus rings.

Icons · 8 px gap

Optional leading or trailing icon slot. Decorative icons use aria-hidden="true".

Touch targets · Small buttons

Medium and Large meet the 44×44 px minimum natively. On touch devices, wrap Small buttons in .ds-btn-hit.ds-btn-hit--sm to expand the invisible hit area without changing visual size.

Inspect on a touch viewport — the wrapper adds ≥ 44 px tap area via ::after on (pointer: coarse).

for everyone. by default.

Accessibility is not a polish pass — it is how Jemm earns trust. Design for clarity, contrast, keyboard flows, and screen readers from the first wireframe.

Color & contrast

Emerald backgrounds use white type with a deep green outline when body-sized. Mint and light support tints use emerald or accent type with an outline stroke. Deep green, steel, ink, and charcoal backgrounds use neon green or white. Neon backgrounds use deep green or charcoal — never white.

Outline typography

When any pairing scores below 4.5:1, add a 1px stroke in the darker of the two colors — deep green on mint, coconut, and emerald fills. Display headlines can use hollow outline type instead.

Focus & keyboard

Every interactive control has a visible focus ring using --primary-ring. Tab order follows visual hierarchy; no keyboard traps in menus or modals.

Motion & media

Respect prefers-reduced-motion. Provide text alternatives for icons and logos. Never rely on color alone to convey state.

Language & UI

Use plain language, descriptive labels, and sufficient touch targets (44×44px minimum). Announce route changes and form errors to assistive tech.
